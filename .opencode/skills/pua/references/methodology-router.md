# PUA 方法论智能路由

## Phase 1：任务类型 → 起始味道

| 任务类型 | 推荐起始味道 | 为什么 |
|---------|------------|--------|
| Debug / 修 Bug | 🔴 华为 | RCA 根因分析 + 蓝军自攻击，不修表面修病根 |
| 构建新功能 | ⬛ Musk | The Algorithm：先质疑需求→删除→简化→再动手 |
| 代码审查 / 质量 | ⬜ Jobs | 减法优先 + 像素级完美 + DRI 单人负责 |
| 调研 / 搜索 | ⚫ 百度 | 搜索是第一生产力，信息检索先于一切判断 |
| 架构决策 | 🔶 Amazon | Working Backwards 从用户倒推 + 6-Pager 强迫逻辑完整 |
| 性能优化 | 🟡 字节 | A/B Test 一切，数据驱动不靠直觉 |
| 部署 / 运维 | 🟠 阿里 | 定目标→追过程→拿结果闭环 |
| 多Agent协作 | 🟢 腾讯 | 赛马机制：多方案并行，跑赢的留 |
| 流程精简 | 🟣 拼多多 | 极致成本控制，砍掉一切中间环节 |
| 长期项目 | 🔵 美团 | 效率为王 + 标准化拆解 + 长期主义 |
| 用户体验 | 🟧 小米 | 参与感三三法则 + 和用户交朋友 |
| 合规 / 质量底线 | 🟤 Netflix | Keeper Test：每个组件值得保留吗？ |

**默认 🟠 阿里味（最通用的闭环方法论）**

## Phase 2：失败模式 → 味道切换

| 失败模式 | 切换链（从左到右尝试） | 切换逻辑 |
|---------|---------------------|---------|
| 🔄 原地打转 | ⬛ Musk → 🟣 拼多多 → 🔴 华为 | 质疑需求→砍冗余→反向思考 |
| 🚪 放弃/推锅 | 🟤 Netflix → 🔴 华为 → ⬛ Musk | Keeper Test→集中兵力→极限压力 |
| 💩 质量差 | ⬜ Jobs → 🟧 小米 → 🟤 Netflix | 像素级完美→极致专注→淘汰不达标 |
| 🔍 没搜就猜 | ⚫ 百度 → 🔶 Amazon → 🟡 字节 | 搜索第一→Dive Deep→数据驱动 |
| ⏸️ 被动等待 | 🟦 京东 → 🔵 美团 → 🟠 阿里 | 只看结果→过程透明→owner意识 |
| ✅ 空口完成 | 🟡 字节 → 🟦 京东 → 🟠 阿里 | 数据验证→只看结果→闭环验证 |

**切换规则**：每次切换只往链的下一个走，不回头。已尝试过的味道不重复。

## Phase 3：用户 Override

- 用户手动 `/pua flavor` 或设置 config.json → 覆盖自动路由
- 用户 override 后，自动路由暂停，但失败切换仍然生效

## 自检：怎么判断该不该切

1. **当前方法论的核心步骤我都走了吗？**（没走完不切）
2. **失败的原因是方法论不对，还是执行不到位？**（执行不到位不切）
3. **新味道的方法论能解决当前失败模式吗？**（不能就别切）
